## Описание задания

В современном мире данных эффективный сбор и интеграция информации из различных источников являются фундаментом для принятия обоснованных решений и успешного ведения бизнеса. Компании, стремясь к конкурентоспособности, активно используют технологии веб-скрапинга и взаимодействия с API для автоматизации процесса сбора данных, обеспечения их актуальности и полноты.
В данном проекте мы на реальных данных попрактиковались в том, как осуществляется веб-скрапинг различных сайтов для извлечения необходимой информации, а также в работе с API популярных сервисов для получения структурированных данных. Мы научились применять современные инструменты и библиотеки, отражающие различные этапы DE, обрабатывать и очищать полученную информацию, а также интегрировать данные из различных источников для последующего использования их в аналитических и бизнес-задачах.


## Этапы проекта

-	Выбор источников данных
-	Формулировка бизнес-задачи, которую могла бы хотеть решить компания/платформа, на основе собираемых данных
-	Обоснование сформулированной бизнес-задачи
-	Извлечение данных путём работы с API
-	Извлечение данных путём скрапинга и парсинга
-	Первичная обработка данных (EDA)
-	Предложения гипотез
-	Получение и агрегация результатов, финальные выводы


## Бизнес-задача

Провести комплексный анализ музыкальной индустрии для оптимизации процессов создания и продвижения музыкальных треков с целью максимизации их коммерческого потенциала. То есть анализ различных характеристик музыкальных треков и их исполнителей на основе данных, взятых с крупных музыкальных платформ и затем анализ их зависимостей. Наша цель в том, чтобы выяснить какие метрики могут повлиять на успех трека и ожидаемый бизнес-результат - это снижение рисков при создании и продвижении музыкального контента за счет data-driven подхода к принятию решений.


## Описание датасетов

### Apple
17809 rows × 25 columns
Взят с сайта « https://music.apple.com/us/new »
Используем requests


### Genius
20670 rows × 11 columns
Взят с сайта «https://genius.com/»
Используем requests + api


### Musicbrainz
36327 rows × 34 columns
Взят с сайта «https://musicbrainz.org/»
Используем requests + bs4


### Spotify+Artist
74245 rows × 28 columns
Взят с сайтов « https://open.spotify.com/» и «https://kworb.net/spotify/»
Получение данных состоит из 3 этапов:
1.  Парсинг с api данных треков с сайта « https://open.spotify.com/»
Используем api
2.  Парсинг артистов и их данных с сайта и «https://kworb.net/spotify/»
Используем  selenium как дополнительный инструмент
3.  Парсинг треков для конкретных артистов с сайта «https://kworb.net/spotify/»
Используем requests + bs4. так же реализована многопоточность, как дополнительный инструмент,  для более эффективного парсинга
Так же для всего парсинга данных Spotify+Artist реализовано логирование как дополнительный инструмент



## Описание столбцов в итоговом датасете:

- `artist` - имя артиста (object)
- `followers` - количество подписчиков у артиста (int64)
- `artist_popularity` - популярность артиста (показатель spotify) (int64)
- `album_title` - название альбома (object)
- `album_release_year` - год выпуска альбома (int64)
- `number_of_tracks` - количество треков в альбоме (int64)
- `track_number_in_album` - номер трека в альбоме (int64)
- `track_popularity` - популярность трека (показатель spotify) (int64)
- `explicit` - не цензурный (1 - да, 0 - нет) (int64)
- `as_feature_tracks` - количество треков, в которых данный исполнитель приглашенный (float64)
- `as_feature_streams` - прослушивания треков, в которых данный исполнитель приглашенный (float64)
- `solo_tracks` - количество треков, в которых данный исполнитель единственный (float64)
- `solo_streams` - прослушивания треков, в которых данный исполнитель единственный (float64)
- `as_lead_tracks` - количество треков, в которых данный исполнитель главный (float64)
as_lead_streams` - прослушивания треков, в которых данный исполнитель главный (float64)
- `total_tracks` - количество треков у исполнителя (float64)
- `total_streams` - количество прослушиваний у исполнителя (float64)
- `track_daily_listening` - ежедневные прослушивания трека (int64)
- `track_total_listening` - общие прослушивания трека (int64)
- `track_title` - название трека (object)
- `track_genre` - жанр трека (object)
- `track_release_date` - полная дата выпуска трека (object)
- `price` - цена трека в iTunes Store (float64)
- `is_single` - является ли трек первым в альбоме (bool)
- `has_feat_in_title` - упоминаются ли другие исполнители (int64)
- `title_length` - длина названия в символах (int64)
- `title_word_count` - количество слов в названии (int64)
- `is_live_comm` - живая версия или нет (bool)
- `is_acoustic_comm` - акустическая версия или нет (bool)
- `is_cover_comm` - кавер версия или нет (bool)
- `content_type` - тип трека (object)
- `featured_artists` - участвующие исполнители (object)
- `views` - количество просмотров страницы на Genius (int64)
- `duration_min_sec` - продолжительность трека (в минутах и секундах) (object)
- `artist_type` - тип исполнителя (object)
- `artist_gender` - пол исполнителя (object)
- `has_features` - наличие приглашенных артистов (int64)
- `is_demo` - демо версия или нет (int64)


Новые(столбцы, добавленные в ходе работы на основе имеющихся столбцов):
- `track_release_day` - день выпуска трека (int32)
- `track_release_month` - месяц выпуска трека (int32)
- `track_release_year` - год выпуска трека (int32)
- `duration_sec` - продолжительность трека (в секундах) (int32)
- `is_pop` - относится ли к какому-нибудь pop жанру(bool)
- `is_hit_song` - популярный трек или нет (bool)
- `artist_avg_popularity` - средняя  популярность трека по всем песням артиста (float64)
- `track_relative_popularity` -  популярность трека / artist_avg_popularity (float64)
- `release_season` - время года когда вышел трек (object)
- `track_age` - возраст трека (int32)


## Пропуски:

- `featured_artist` - 8093 пропусков, так как большинство треков не имеют соавторов, заполнили их ‘without_featured_artist’

В столбцах `as_feature_tracks`, `as_feature_streams`, `solo_tracks`, `solo_streams`, `as_lead_tracks`, `as_lead_streams`, `total_tracks`, `total_streams` есть 70 пропусков в одинаковых строках, они связаны с отсутствием данных по одному артисту, так как это достаточная маленькая часть от всех строк, было принято решение убрать строки по этому исполнителю


- `artist_gender` - 2533 пропуска, так как есть не индивидуальные исполнители, а группы, заполнили их ‘not solo’
